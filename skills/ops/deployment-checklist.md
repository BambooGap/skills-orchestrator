# 部署检查清单 Skill

## 概述

每次生产部署前必须过一遍此清单。**没有跳过项，只有"不适用"**。
目标：让每次部署都可预测、可回滚、有监控覆盖。

---

## 部署前（T-1 天）

### 代码准备
- [ ] 所有 PR 已合并，分支已删除
- [ ] `main` 分支 CI 全绿
- [ ] 部署 PR 描述包含：改了什么 + 如何测试 + 回滚步骤
- [ ] 数据库迁移脚本已审查（两人审查制）

### 环境检查
- [ ] Staging 已部署并验证通过
- [ ] Staging 数据是否代表性（非空库）
- [ ] 新配置/环境变量已在 Staging 验证

### 通知
- [ ] 通知相关团队（前端/客服/运营）
- [ ] 高流量时段不部署（参考流量监控）

---

## 数据库变更顺序（分离部署）

**强制规则**：代码部署和数据库变更分开，数据库先行。

```
# 正确顺序（以新增字段为例）
Step 1: 部署 DB 迁移（添加可为 NULL 的新列）
Step 2: 部署代码（同时写旧列和新列）
Step 3: 回填旧数据
Step 4: 加 NOT NULL 约束（如需要）
Step 5: 部署代码（只写新列）
Step 6: 删除旧列（下下个版本）
```

---

## 部署中

### 滚动发布策略
- [ ] 先发 1 台（金丝雀），观察 5 分钟
- [ ] 检查错误率是否升高（对比基线）
- [ ] 检查响应时间是否退化
- [ ] 继续发其余实例

### 关注指标
```
必看：
  - HTTP 5xx 错误率（阈值：< 0.1%）
  - P99 响应时间（阈值：< 正常值 × 1.5）
  - 数据库连接数（阈值：< 80% 最大连接）

按需看：
  - 队列积压量
  - 内存/CPU 使用率
  - 第三方 API 错误率
```

---

## 部署后（T+30 分钟）

- [ ] 核心用户流程人工验证（登录/主要功能/支付）
- [ ] 检查日志中是否有新的 ERROR 级别日志
- [ ] 告警规则已生效（触发一次测试告警）
- [ ] 数据库慢查询监控正常

---

## 回滚决策树

```
发现问题 →
  ├── 是否影响核心功能？
  │   ├── 是 → 立即回滚，不等待修复
  │   └── 否 → 评估修复时间
  │           ├── < 30 分钟 → 前进修复
  │           └── > 30 分钟 → 回滚
  └── 回滚步骤
      1. 执行: kubectl rollout undo deployment/app
         或: git revert + 快速部署
      2. 通知团队
      3. 记录事故时间线
```
