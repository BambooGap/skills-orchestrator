# 中文代码审查 Skill

## 概述

本 skill 定义适配国内团队的代码审查标准，涵盖命名规范、注释规范、PR 描述要求、代码异味识别。适用于所有需要进行代码审查的场景，确保代码质量和团队协作效率。

---

## 命名规范检查清单

### 变量命名

| 检查项 | 规范 | 不合规示例 | 合规示例 |
|--------|------|------------|----------|
| 命名风格 | 小驼峰或蛇形，项目统一 | `UserName` / `user_name` 混用 | `userName` 或 `user_name` |
| 语义明确 | 变量名表达用途 | `temp` / `data` / `x` | `processedOrders` / `totalAmount` |
| 避免缩写 | 除非是通用缩写 | `usrNm` / `amt` | `userName` / `amount` |
| 常量命名 | 全大写蛇形 | `MAX_VALUE` | `MAX_RETRY_COUNT` |
| 私有变量 | 单下划线前缀 | `__private`（双下划线） | `_internalState` |

### 函数命名

| 检查项 | 规范 | 不合规示例 | 合规示例 |
|--------|------|------------|----------|
| 动词开头 | 表达动作 | `data()` / `info()` | `getData()` / `getInfo()` |
| 参数名明确 | 参数名表达含义 | `func(a, b)` | `calculateTotal(price, quantity)` |
| 返回值暗示 | 名字暗示返回类型 | `process()` | `processAndReturnResult()` |
| 避免否定 | 用正向表达 | `isNotValid()` | `isValid()` |

### 类命名

| 检查项 | 规范 | 不合规示例 | 合规示例 |
|--------|------|------------|----------|
| 大驼峰 | 类名首字母大写 | `user_service` | `UserService` |
| 名词结尾 | 类是事物或概念 | `DoSomething` | `OrderProcessor` |
| 避免冗余 | 不重复父类含义 | `UserUserInfo` | `UserInfo` |
| 接口命名 | I 前缀或 able 后缀 | `InterfaceUser` | `IUserService` 或 `Payable` |

### 文件命名

| 检查项 | 规范 | 不合规示例 | 合规示例 |
|--------|------|------------|----------|
| 模块名 | 小写蛇形 | `UserService.py` | `user_service.py` |
| 测试文件 | test_ 前缀 | `service_test.py` | `test_user_service.py` |
| 配置文件 | 小写蛇形或 yaml | `Config.JSON` | `config.yaml` |

## 注释规范

### 必须写注释的情况

1. **复杂算法**：解释算法思路和关键步骤
2. **业务规则**：解释业务逻辑的背景和原因
3. **公共接口**：API 文档注释（参数、返回值、异常）
4. **临时方案**：标记 TODO/FIXME，说明后续计划
5. **边界条件**：解释特殊处理的边界情况
6. **性能优化**：解释为什么这样写能提升性能

### 不需要写注释的情况

1. **自解释代码**：命名清晰、逻辑简单的代码
2. **重复文档**：已有文档说明的内容
3. **显而易见**：`i += 1` 这种无需解释
4. **注释掉的代码**：应该删除，不是注释

```python

"""
```

## PR 描述要求

### 必填内容

1. **变更类型**：feat/fix/refactor/docs/test/chore
2. **变更说明**：描述本次变更的主要内容
3. **关联 Issue**：链接相关 issue 编号
4. **测试情况**：测试覆盖率、测试结果

### 可选内容

1. **风险评估**：影响范围、回滚方案
2. **截图/演示**：UI 变更的截图或 GIF
3. **依赖变更**：新增/删除的依赖包

### PR 标题格式

```
<type>: <简短描述>

示例：
feat: 添加用户注册功能
fix: 修复订单金额计算错误
refactor: 抽取用户服务为独立模块
```

## 代码异味识别

### 常见代码异味清单

| 异味 | 表现 | 影响 | 建议 |
|------|------|------|------|
| **过长函数** | 函数超过 50 行 | 难理解、难测试 | 拆分为多个小函数 |
| **重复代码** | 相似代码块多处出现 | 维护成本高 | 抽取公共函数 |
| **过大类** | 类超过 200 行或 10 个方法 | 职责不清 | 拆分为多个类 |
| **参数过多** | 函数参数超过 4 个 | 调用复杂 | 使用对象封装参数 |
| **魔法数字** | 硬编码数字无说明 | 意义不明 | 定义常量并命名 |
| **嵌套过深** | if/for 嵌套超过 3 层 | 逻辑混乱 | 提取函数或使用卫语句 |
| **全局变量** | 使用全局状态 | 难追踪、难测试 | 改为参数传递 |
| **注释过多** | 代码大部分是注释 | 干扰阅读 | 简化代码或删除冗余注释 |

### 检查方法

1. **静态分析**：使用 pylint/flake8 等工具
2. **人工审查**：逐行阅读关键模块
3. **历史对比**：对比变更前后的代码
4. **测试覆盖**：难以测试的代码往往有问题

## 审查报告输出格式

### 审查结果表格

| 文件 | 行号 | 问题类型 | 严重程度 | 描述 | 建议 |
|------|------|----------|----------|------|------|
| user_service.py | 45 | 命名 | 中 | 变量名 `temp` 不明确 | 改为 `processedUsers` |
| order.py | 120 | 异味 | 高 | 函数过长（80行） | 拆分为 `validateOrder` 和 `processPayment` |
| config.py | 30 | 注释 | 低 | 缺少业务规则说明 | 添加折扣规则注释 |

### 严重程度定义

| 级别 | 说明 | 处理方式 |
|------|------|----------|
| **高** | 影响功能或安全 | 必须修复，阻塞合并 |
| **中** | 影响可读性或维护性 | 建议修复，可后续处理 |
| **低** | 风格问题或建议 | 可忽略，或后续优化 |

## 审查流程

1. **理解变更意图**：阅读 PR 描述和关联 Issue
2. **整体浏览**：快速浏览所有变更文件
3. **逐文件审查**：按重要性排序，逐文件检查
4. **记录问题**：使用表格格式记录发现的问题
5. **给出建议**：每个问题给出具体修复建议
6. **正向反馈**：指出做得好的地方，鼓励作者
