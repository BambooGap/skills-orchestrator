# 任务估时规范 Skill

## 概述

估时的目标不是"准确预测"，而是**暴露不确定性，驱动早期沟通**。
适用场景：Sprint 规划、项目排期、新功能评估。

---

## 核心原则

**不要给你没有拆解的任务估时。**

大任务（> 2 天）的估时误差通常是 2-5 倍。拆解到叶节点（每个 ≤ 1 天）后，总和的误差通常在 1.5 倍以内。

---

## 拆解方法

### 拆解到可执行的叶节点

```
差的拆解：
  实现用户认证功能（5天）

好的拆解：
  ├── 设计 DB Schema（users, sessions 表）  0.5天
  ├── 实现注册接口（POST /users）           1天
  │   ├── 输入验证
  │   ├── 密码哈希
  │   └── 发送验证邮件
  ├── 实现登录接口（POST /sessions）        0.5天
  ├── JWT Token 生成与验证中间件            1天
  ├── 实现登出接口                          0.25天
  └── 集成测试                              0.75天
  合计：4天（差了但可控）
```

**检查**：每个叶节点应该能在一天内完成，且结束状态清晰可验证。

---

## 三点估计法

每个任务给三个数字：

```
乐观（O）：顺利情况下
期望（M）：最可能情况
悲观（P）：遇到阻塞的情况

加权平均（PERT）= (O + 4M + P) / 6
标准差 σ = (P - O) / 6
```

**示例**：

| 任务 | O | M | P | PERT | σ |
|------|---|---|---|------|---|
| 实现注册接口 | 0.5 | 1 | 2 | 1.08 | 0.25 |
| JWT 中间件 | 0.5 | 1 | 3 | 1.25 | 0.42 |
| 集成测试 | 0.25 | 0.75 | 2 | 0.83 | 0.29 |

---

## 显式列出不确定性

```
## 估时：实现支付集成（Stripe）

PERT 估计：3.5 天

### 已知的不确定性
- Stripe 测试账号申请需要 1-2 天审核（外部依赖）
- Webhook 本地调试环境搭建经验不足（+0.5 天缓冲）
- 退款流程复杂度未知（+？）

### 前提假设
- 假设 Stripe API 文档清晰（如果不是 +1 天）
- 假设测试环境 Stripe 账号已经存在

### 给出的区间
- 乐观：2.5 天
- 期望：3.5 天
- 最差：6 天（如果 Stripe 审核慢 + webhook 复杂）
```

## 估时的常见陷阱

| 陷阱 | 后果 | 预防 |
|------|------|------|
| 只估"写代码"时间 | 忽略调试/review/部署 | 加 20-30% buffer |
| 假设"没有干扰" | 会议、slack 消耗 40% 时间 | 按实际工作日计算 |
| 第一次做新技术 | 学习曲线被低估 | 明确标注"首次使用，乘以 2" |
| 集成依赖第三方 | 等待时间不可控 | 列入不确定性清单 |
| 忘记测试时间 | 功能完成≠任务完成 | 测试时间单独列项 |

---

## 估时后的跟踪

```
每天结束时记录：
- 今天完成了什么（与计划的差距）
- 剩余工作量重新估计
- 如果超出 20%，立即沟通（不要等到最后）
```

**"进展正常"是最危险的状态**：直到截止日才发现问题。

## 与其他 Skill 的关系
