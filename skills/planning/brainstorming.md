# 结构化头脑风暴 Skill

## 概述

本 skill 提供一套结构化的头脑风暴方法论，帮助在开始编码前充分展开问题，探索多种解决方案，避免过早收敛到第一个想到的方案。

---

## 何时使用

- 遇到设计决策（方案 A 还是方案 B？）
- 功能边界不清晰时
- 遇到难以复现的 bug
- 需要估算任务复杂度前
- 多人协作时对齐思路

---

## 头脑风暴流程

### 第一步：问题定义（5 分钟）

在展开之前，先写下：

```
问题：[用一句话描述实际要解决的问题]
约束：[不可更改的限制，如技术栈、时间、兼容性]
目标：[成功的标准是什么]
```

**常见陷阱**：直接跳到解决方案而没有定义清楚问题。

### 第二步：发散（10 分钟）

列举至少 5 个方案，**不筛选**，包括看起来荒谬的：

| 方案 | 核心思路 | 预期效果 |
|------|---------|---------|
| 方案 1 | ... | ... |
| 方案 2 | ... | ... |
| 方案 3 | ... | ... |
| 方案 4（激进）| ... | ... |
| 方案 5（最简）| ... | ... |

**技巧**：
- "如果时间无限，我会怎么做？" → 理想方案
- "如果只有 1 小时，我会怎么做？" → 最小方案
- "如果完全推翻现有设计，我会怎么做？" → 颠覆方案

### 第三步：收敛（10 分钟）

用矩阵筛选：

| 方案 | 实现难度(1-5) | 维护成本(1-5) | 风险(1-5) | 覆盖场景 | 总分 |
|------|-------------|-------------|---------|---------|------|
| 方案 1 | | | | | |
| 方案 2 | | | | | |

分数越低越好。选总分最低且覆盖核心场景的方案。

### 第四步：压力测试（5 分钟）

对选定方案提问：

1. **边界场景**：这个方案在极端情况下会发生什么？
2. **失败模式**：如果这个方案失败，怎么回滚？
3. **隐藏假设**：这个方案依赖什么前提条件？
4. **6 个月后**：6 个月后这段代码还好维护吗？
