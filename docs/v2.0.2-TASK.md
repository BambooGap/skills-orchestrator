# v2.0.2 工程质量收口任务单

基于 skills-orchestrator v2.0.1 做 v2.0.2 工程质量收口，不新增大功能，不改变公开 CLI 行为。

必须按以下顺序执行，不允许一次性大重构。

## 第一阶段：补测试和报告

1. 新增 `reports/TEST_REPORT_v2.0.2.md`，不再覆盖根目录 `TEST_REPORT.md`。
2. 根目录 `TEST_REPORT.md` 改为索引，指向历史报告。
3. 报告字段固定包含 commit、tag、Python、OS、命令、结果、时间。
4. 补 import 命令测试，必须 mock 网络，不允许真实访问 GitHub。
5. import 测试覆盖：
   - GitHub blob URL 转 raw URL
   - raw.githubusercontent.com 白名单
   - 非 GitHub 域名拒绝
   - timeout
   - 404
   - frontmatter YAML 解析失败
   - 空内容
   - 无标题且无 summary
   - 危险文件名
   - 路径穿越
6. 注意：无 frontmatter 的 markdown 不是非法，只要当前逻辑允许 metadata 推断，就必须保持兼容。
7. 补 MCP CLI 测试：
   - mcp-test get_skill nonexistent 必须返回非 0。
   - stderr 必须包含明确错误。
   - MCP executor 不吞异常。

## 第二阶段：机械拆分 main.py

1. 当前已有 `skills_orchestrator/cli/init_cmd.py` 和 `cli/helpers.py`，但 main.py 仍保留重复实现。
   - **实际发现**：cli/ 目录不存在，所有逻辑仍在 main.py 中。
2. 先只迁移 init 命令。
3. main.py 使用 `cli.add_command(init)` 注册。
4. 删除 main.py 中重复 init 实现。
5. 保证 init 行为完全不变。
6. 测试通过后，再考虑拆 import、pipeline、sync、mcp 命令。
7. 不允许一次性全拆。

## 第三阶段：安全边界

1. subprocess PATH 不要只写死 `/usr/bin:/bin`。
2. 默认安全 PATH 建议包含：`/usr/local/bin:/opt/homebrew/bin:/usr/bin:/bin`
3. 允许通过 `SKILLS_ORCHESTRATOR_SAFE_PATH` 覆盖。
4. 输出路径防逃逸必须按命令分类：
   - init/import 默认限制在 project root 或 skills dir
   - sync hermes 允许 `~/.hermes/skills`
   - sync openclaw 允许 `~/.openclaw`
   - 不允许用一个 cwd root 一刀切

## 第四阶段：配置和策略收口

1. 如果 tag_categories.yaml 已经配置化，必须保证 wheel 安装后可读取。
2. 将配置放入 `skills_orchestrator/config/tag_categories.yaml`。
3. 使用 `importlib.resources` 读取。
4. `pyproject.toml` 必须包含 package data。
5. effective_policy 只能有一个权威计算入口。
6. resolver.py 不应继续手写 forced skill 判断。
7. 所有 forced / zone require / skill load_policy 统一走 `compiler/policies.py`。

## 每完成一个阶段必须运行

- `pytest tests/ -v`
- `ruff check .`
- `ruff format --check .`
- `python -m build`
- `skills-orchestrator --help`
- `skills-orchestrator init --help`
- `skills-orchestrator mcp-test --help`

## 最终提交要求

1. 不新增大功能。
2. 不破坏现有 CLI。
3. 不改变兼容的 Markdown 解析行为。
4. 不引入两套命令实现。
5. 不允许测试绿但 wheel 安装后配置文件读不到。
